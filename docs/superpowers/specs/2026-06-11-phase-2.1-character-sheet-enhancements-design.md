# Phase 2.1 ‚Äî Character Sheet Enhancements ‚Äî Design

**Implemented:** 2026-06-11 (executed autonomously; live-API paths to be verified by the user ‚Äî see "Live-API assumptions").

Builds on Phase 2 (`docs/superpowers/plans/2026-06-11-phase-2-character-sheet.md`). Two bug fixes (shipped) plus three features and an information-architecture reorg. Rolling stays Phase 3; spell data arrives in Phase 3 (popups are built to accept spells later). Decisions follow how desktop PF2e groups and presents these things.

---

## 1. Hero-point pips ‚Äî fix (shipped: `8a71e9c`)

**Symptom:** hero points showed nothing at value 0 (probe confirmed `{ value: 0, max: 3 }`, so data was fine).

**Root cause:** `src/styles/tailwind.css` disables Tailwind preflight and adds a scoped reset `#pf2e-mobile-companion-root button { ‚Ä¶ border: none; }`. That rule is *unlayered* with id-specificity, so it outranks the layered `.border` utility ‚Äî bordered `<button>`s lose their border. Empty pips were `bg-transparent` + a (dead) border ‚Üí invisible; at 0 hero points all three vanished. (Bordered non-buttons are unaffected: `.border` keeps `border-style: solid`.)

**Fix:** pips render as a background fill ‚Äî `bg-amber-400` (filled) / `bg-zinc-600` (empty), no border. Applies to hero / dying / wounded. Tap a pip to gain/spend (existing `onAdjust`).

## 2. Shield HP edit ‚Äî fix (shipped: `b6d1581`)

**Root cause:** `system.attributes.shield` is *derived* from the equipped shield item each prepare cycle, so `actor.update({"system.attributes.shield.hp.value": n})` was discarded.

**Fix:** `setShieldHp` resolves the shield item via `system.attributes.shield.itemId` and updates `item.update({"system.hp.value": n})` ‚Äî the same item-update pattern equip/invest use.

## 3. Tap-for-info popups

Tapping a **feat**, an **inventory item**, or an **effect** opens a bottom-sheet showing PF2e's own detail: name, type/level, traits, action cost (feats) or bulk¬∑price¬∑quantity¬∑usage (items), and the description. **Spells** are deferred to Phase 3 (no spell data in the sheet yet); **conditions** keep their existing manager modal.

- **Mechanism (lazy):** on tap, read the live document `actor.items.get(id)`, extract fields via a pure `buildItemDetail(itemLike)`, then render in the shared `Modal`. Descriptions stay out of the snapshot (`CharacterView`) ‚Äî only fetched on demand.
- **Description:** best-effort enriched via Foundry's `TextEditor.enrichHTML` (resolves `@UUID`/trait links); falls back to the raw HTML string (as `BioPanel` already does) if enrichment isn't reachable. Inline-roll/link interactivity is Phase 3/6.

## 4. Bio reorg ‚Äî Hybrid IA (user-chosen)

Desktop PF2e keeps Biography as prose only, with lineage/languages in the main Character view and a dedicated Proficiencies area. Mapped onto our tab bar:

- **New "Profs" sub-tab** ‚Äî weapon + armor proficiencies. (Class DC stays in Vitals; spell DCs arrive with Phase 3.)
- **Vitals** gains a top **Character** section (ancestry / heritage / background / class / deity / size) and a **Languages** section.
- **Bio** = prose only (appearance / backstory).

Pure presentation change ‚Äî reuses existing `BioView` data; no mapper change. Tab order: Vitals ¬∑ Skills ¬∑ Items ¬∑ Feats ¬∑ **Profs** ¬∑ Bio.

## 5. Modifier breakdowns

Tapping a number ‚Äî **AC, saves, perception, skills, class DC** (in the Vitals and Skills panels) ‚Äî opens a popup listing each contributing modifier (label + signed value) and the total, mirroring PF2e's modifier tags. Header chips (AC/Per) stay quick-reference for now.

- **Data:** the mapper extracts each statistic's enabled `.modifiers` ‚Üí `{ label, value }`. A defensive reader checks `stat.modifiers ?? stat.check?.modifiers ?? []` (covers both `StatisticModifier` and `Statistic` shapes) and filters `enabled !== false`. A missing array yields empty parts ‚Üí the popup shows the total only (never crashes).
- **Phase-3 seam (no conflict with rolling):** no rolling code here. The `BreakdownModal` takes an optional `onRoll?` that is **omitted now** (display-only); Phase 3 supplies it to add a "Roll" button to the same popup. Phase 3 rolling will call the live statistic objects (`actor.skills[x].roll()`), independent of this display data.

## Components & data layer

- **Reuse:** `parts/Modal.tsx` (bottom sheet) for both new popups.
- **New components:** `BreakdownModal.tsx` (#5), `DetailModal.tsx` (#3), `ProfsPanel.tsx` (#4).
- **New foundry helpers:** `buildItemDetail` (pure) + `ItemDetailView`/`ItemDetailLike` types; `enrichHtml(html)` best-effort wrapper (`src/foundry/enrich.ts`).
- **View additions (all optional, graceful):** `ModPartView { label; value }`; `breakdown?: ModPartView[]` on `SaveView`, `SkillView`, `ClassDcView`, and `DefensesView.perception`; `acBreakdown?: ModPartView[]` on `DefensesView`. `id` added to `EffectView`/`EffectLike` so effect chips can open detail.
- **Structural (`CharacterLike`) additions:** optional `modifiers?: ModifierLike[]` on `attributes.ac`, `saves[slug]`, `perception`, `proficiencies.classDCs[key]`; optional `modifiers`/`check.modifiers` on `SkillLike`. `ModifierLike { label; modifier; enabled?; type? }`.
- **No new mutations** ‚Äî #3 and #5 are read-only.

## Live-API assumptions (verify; correct any path on return)

- **Breakdowns:** `system.attributes.ac.modifiers`, `system.saves[slug].modifiers`, `system.perception.modifiers`, `system.proficiencies.classDCs[key].modifiers`, and per-skill `actor.skills[slug].modifiers` (or `.check.modifiers`) ‚Äî each entry `{ label, modifier, enabled }`.
- **Detail:** `item.system.description.value` (HTML), `system.traits.value`, `system.level.value`, `system.actionType/actions` (feats), `system.quantity/bulk/price/usage` (items).
- **Enrichment:** the `TextEditor.enrichHTML` location on Foundry v14 (global vs. `foundry.applications.ux`). Falls back to raw HTML if not found.
