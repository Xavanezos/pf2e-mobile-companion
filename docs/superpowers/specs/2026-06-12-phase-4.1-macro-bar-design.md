# Phase 4.1 — Macro bar (Map tab) — Design

**Status:** Specced and approved in brainstorming (2026-06-12); ready for an implementation plan. A single, self-contained slice. Live-API paths are grounded against the vendored Foundry types in the cloned PF2e checkout (`E:/React Projects/pf2e/types/foundry`); confirm in play (see **Live-API assumptions**).

The Foundry **hotbar macros bar** — explicitly deferred out of Phase 4 ("mirror only") — lands here as a horizontally-scrolling strip **permanently pinned at the bottom of the Map tab**, above the global tab nav. It mirrors the player's hotbar: every populated slot, flattened, in slot order. Tapping a button **executes the macro** exactly as clicking the desktop hotbar would — chat macros post their message, script macros run in the player's client with their permissions. As everywhere else in the app, Foundry owns execution and permissions; we only render and dispatch.

**Why the Map tab:** macros are most useful while watching the fight and moving your token (decided in brainstorming). The Map tab (today a bare `Placeholder`, "Battle Map — coming in Phase 7") gains a real `MapTab` host now; until Phase 7 builds the map, the area above the bar is the existing placeholder and the bar is fully usable on top of it.

**Out of scope (YAGNI):** assigning / rearranging / removing macros from mobile (arrange the hotbar on desktop); a hotbar-pages UI (we flatten all pages); long-press macro detail; showing the bar on any tab other than Map; token-scoped execution (no canvas placeable until Phase 7).

---

## Decisions (approved)

1. **Placement: permanently pinned at the bottom of the Map tab**, above the tab nav (not global, not collapsible, not a section of the Actions tab). A new `MapTab` hosts it.
2. **Contents: all populated hotbar slots across all 5 pages, flattened** into one horizontal scroll strip, in ascending slot order. No page-flip affordance — swipe to scroll.
3. **Button style: the macro's icon + a tiny truncated name**; the whole button is the tap target → execute. (Not icon-only.)
4. **Execution passes `{ actor }` scope** — the app's active actor — so actor-aware macros work without a canvas selection. **Token scope is omitted** (no canvas placeable on mobile until Phase 7); canvas-dependent macros may no-op/error, surfaced as a Foundry toast. Strictly better than a bare `macro.execute()` on a canvas-less client.
5. **Read-only mirror.** The bar reflects the live hotbar and re-preps when it changes on desktop; it never writes to the hotbar.
6. **No new chat infrastructure.** Chat macros post real PF2e messages → already surfaced by the existing Slice-1 chat feed (Chat tab + own-result toast).

## Architecture & data flow

Follows the established pattern exactly (pure sync mapper → version-bumped hook → guarded action → thin UI), the same shape as `useToggles`/`toggles.ts` and `useStrikes`/`strikeActions.ts`.

```
Map tab  (new MapTab: map area [Phase-7 placeholder, flex-1]  +  MacroBar [pinned, shrink-0])
   │
   ├─ useHotbar()  → buildHotbarView(user)        [pure, SYNC]
   │      reads game.user.hotbar  (Record<slot, macroId>)
   │      → sort by numeric slot → game.macros.get(id)
   │      → skip dangling (deleted / not visible)
   │      → MacroButtonView[]  { id, slot, name, img, canExecute }
   │      re-preps on  updateUser (slot changes) / updateMacro / deleteMacro
   │
   └─ tap a button
          → src/foundry/macros/hotbar.ts   executeMacro(macroId, actorId)   [guarded]
               re-read game.macros.get(macroId)  + game.actors.get(actorId)
               → macro.execute({ actor })
                        │
                        ▼  chat macro → ChatMessagePF2e ;  script macro → runs in client
                 createChatMessage hook → existing chat feed (ChatTab + ChatToast)
```

- **The mapper is synchronous.** `game.user.hotbar` and `game.macros` are plain in-memory collections — no async prep (unlike spells). So the hook mirrors `useToggles`/`useStrikes`: a `useMemo` invalidated by a version bump, **not** the async `useSpells`.
- **Never hold live objects in React state.** The mapper keeps only the macro **id** + display fields; `executeMacro` re-reads the live `Macro` by id at tap time — same discipline as strikes/spells/toggles.
- **Execution is a side effect.** Running a macro doesn't mutate the bar; the bar changes only when the hotbar/macro Documents do.

### `buildHotbarView(user)` — pure, sync

```ts
// game.user.hotbar : Record<number, string>  (slot 1–50 → macro id)
function buildHotbarView(user: HotbarUserLike): MacroButtonView[] {
  const slots = Object.entries(user.hotbar ?? {})
    .map(([slot, id]) => [Number(slot), id] as const)
    .sort((a, b) => a[0] - b[0]);
  const out: MacroButtonView[] = [];
  for (const [slot, id] of slots) {
    const macro = game.macros?.get(id);
    if (!macro) continue;                       // dangling: deleted / not visible → skip
    out.push({ id, slot, name: macro.name ?? "", img: macro.img ?? null, canExecute: macro.canExecute !== false });
  }
  return out;
}
```

### `executeMacro(macroId, actorId)` — guarded (like `setToggle`)

```ts
async function executeMacro(macroId: string, actorId: string | null): Promise<void> {
  try {
    const macro = game.macros?.get(macroId);
    if (!macro?.execute) throw new Error(`macro ${macroId} not executable`);
    const actor = actorId ? game.actors?.get(actorId) : undefined;
    await macro.execute(actor ? { actor } : {});   // token omitted: no canvas placeable on mobile
  } catch (err) {
    console.error("[pf2e-mobile] executeMacro failed", err);
    ui?.notifications?.error?.("Macro failed — see console.");
  }
}
```

### `useHotbar()` — hook

`useMemo(() => buildHotbarView(game.user), [version])`, with `useFoundryHook` registrations that `bump()` on: `updateUser` (filtered to `doc.id === game.user.id` — fires on slot assign/rearrange/remove), `updateMacro` and `deleteMacro` (name/icon edits + dangling resolution). Returns `null` only if `game.user` is somehow absent (defensive; the app is past `ready`).

### `MacroBar` + `MapTab` — UI

- **`MacroBar`**: a `shrink-0` strip, `flex gap-2 overflow-x-auto` with horizontal scroll. Each button is a vertical icon-over-name cell (~44px icon, `<img>` of `macro.img`, name truncated below); `canExecute === false` → dimmed + `disabled`. Empty hotbar → a centered hint: *"No macros on your hotbar — add them from the desktop."* Buttons follow the Tailwind-v4 button gotchas (solid `bg-*`/`ring-*` fills, not `border`; `justify-*` as needed — see [[styling-gotchas]]).
- **`MapTab`**: `flex flex-col h-full`. The map area is `flex-1 min-h-0` and renders the existing Phase-7 `Placeholder` content for now; `<MacroBar>` sits below it, pinned. (Shell's `<main>` is `overflow-y-auto`; `MapTab` manages its own column so the bar stays put within the tab viewport. Phase 7 fills the map area above the bar without touching the bar.)

## Live-API assumptions (verify in play; correct any path on return)

Grounded against the vendored Foundry types in the PF2e checkout:

1. **`game.user.hotbar`** is `Record<number, string>` — slot (1–50) → macro id (`types/foundry/common/documents/user.d.mts:86`, `hotbar: fields.ObjectField<Record<number, string>>`). Flattening all entries = all pages.
2. **`Macro#execute(scope?)`** accepts `{ actor?: Actor; token?: Token; [k]: unknown }`, returns a `ChatMessage` (chat macros) or the script's return value (`types/foundry/client/documents/macro.d.mts:41-48`). Confirm passing `{ actor }` is honored by actor-aware table macros.
3. **`Macro#canExecute`** (boolean) gates script-macro execution for the current user (`macro.d.mts:32`); **`Macro#img`** is the standard document image (with `Macro#thumbnail` as a getter fallback, `macro.d.mts:35`).
4. **`game.macros.get(id)`** resolves a hotbar id to the live `Macro`, or `undefined` when deleted / not visible to the player → the mapper skips it (no empty gaps).
5. **`updateUser`** fires when the hotbar object changes (assign/rearrange/remove on desktop) → the bar re-preps within ~1s. (`User#assignHotbarMacro`/`getHotbarMacros` exist as the per-page helpers, `user.d.mts:84-123`; we read the raw `hotbar` record to flatten in one pass.)
6. **Canvas-less execution:** with `core.noCanvas` on, `canvas.tokens` is absent, so we cannot supply a `token` placeable and macros that read `canvas.tokens.controlled` fall back to their own `game.user.character` path or fail → toast. Accepted limitation until Phase 7.

## Testing

- **TDD the pure mapper** (Vitest), asserting the structural contract:
  - `buildHotbarView` — ascending slot ordering across pages (e.g. slots `3, 12, 41` → that order); **dangling-skip** (an id with no `game.macros` hit is omitted, no gap); **empty hotbar → `[]`**; `name`/`img`/`canExecute` mapping (incl. `img: null` and `canExecute` default-true when the field is absent).
- **Guarded `executeMacro`, the hook, and `MacroBar`/`MapTab`** are thin glue over live objects — covered by `npm run typecheck` + `npm run build` (test the **production** build, not only `npm run dev` — see [[phase-3-progress]]) and a manual live checklist (Player1 @ mobile width):
  - The bar renders the player's hotbar macros in slot order with correct icons/names; an empty hotbar shows the hint.
  - Tapping a **chat macro** posts its message → appears in Chat + toast; tapping a **script macro** (e.g. an effect/utility) runs and its result/toast appears.
  - A macro that reads the active actor (`scope.actor`) acts on the selected character.
  - Rearranging/removing a slot on desktop updates the bar within ~1s; deleting a macro removes its button (no dangling gap).
  - The bar stays pinned above the tab nav while the map-area placeholder fills the space above it; horizontal scroll works when macros overflow the width.
